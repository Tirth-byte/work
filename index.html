T-03 Enable Row-Level Security (RLS) on ALL tables — company isolation
Tirth Critical Day 1
▼
Why this can't wait
⚠️ Non-negotiable: If company A can ever see company B's QA checklists, you lose both customers and face a lawsuit. Set RLS on Day 1 before writing a single API endpoint. It takes 10 minutes and saves you from a catastrophic data breach.
SQL to Run
-- Enable RLS on all tables ALTER TABLE companies ENABLE ROW LEVEL SECURITY; ALTER TABLE pull_requests ENABLE ROW LEVEL SECURITY; ALTER TABLE checklists ENABLE ROW LEVEL SECURITY; ALTER TABLE feedback ENABLE ROW LEVEL SECURITY; -- Policy: company can only see its own data CREATE POLICY "company_isolation" ON checklists FOR ALL USING (company_id = auth.uid()::uuid); -- Repeat for pull_requests and feedback

T-05 Webhook endpoint: receive PR event → validate signature → push to BullMQ queue
Tirth Critical Day 2
▼
The Golden Rule
MUST respond in <50ms. Receive webhook → validate → push to queue → return 202. ZERO processing in the handler. If you call Claude API here, GitHub will timeout, retry, and you get duplicate jobs.
Code
import { Hono } from 'hono' import { Queue } from 'bullmq' import { verifyWebhookSignature } from './github' const prQueue = new Queue('pr-analysis', { connection: { url: process.env.REDIS_URL } }) app.post('/webhooks/github', async (c) => { // 1. Verify it's really GitHub (HMAC-SHA256) const sig = c.req.header('x-hub-signature-256') const body = await c.req.text() if (!verifyWebhookSignature(body, sig)) { return c.json({ error: 'unauthorized' }, 401) } const event = JSON.parse(body) if (event.action !== 'opened' && event.action !== 'synchronize') { return c.json({ ok: true }) // ignore other events } // 2. Push to queue — do NOT process here await prQueue.add('analyze', { repo: event.repository.full_name, prNumber: event.pull_request.number, prTitle: event.pull_request.title, diffUrl: event.pull_request.diff_url, companyId: 'lookup-by-github-org' }, { jobId: `${event.repository.full_name}-${event.pull_request.number}` }) // 3. Return immediately — NEVER make GitHub wait return c.json({ queued: true }, 202) })

T-07 Integrate Claude API: GitHub diff in → structured QA checklist JSON out
Tirth Critical Day 3 · Morning

Step-by-step

  • Install: pnpm add @anthropic-ai/sdk

Function skeleton
import Anthropic from '@anthropic-ai/sdk' import { z } from 'zod' const ChecklistSchema = z.object({ riskLevel: z.enum(['high', 'medium', 'low']), summary: z.string(), testCases: z.array(z.object({ area: z.string(), // e.g. "Authentication" step: z.string(), // e.g. "Go to /login, enter invalid email" expected: z.string() // e.g. "Error message shown, no token created" })) }) export async function generateChecklist(diff: string) { const client = new Anthropic() const msg = await client.messages.create({ model: 'claude-3-5-haiku-20241022', max_tokens: 2048, system: SYSTEM_PROMPT, // T-08 below messages: [{ role: 'user', content: diff }] }) const raw = JSON.parse(msg.content[0].text) return ChecklistSchema.parse(raw) // throws if AI output is wrong }

T-08 Write the system prompt — this IS your core product, spend 80% of time here
Tirth Critical Day 3 · All Day
▼
Why this matters more than anything
The system prompt is your real product. A bad prompt gives "test the UI" nonsense. A good prompt gives "Go to /checkout. Add item. Remove it. Click Pay — button should stay disabled." Competitors who built generic GPT wrappers have a bad prompt. This is how you win.
Starting prompt template
const SYSTEM_PROMPT = ` You are a senior QA engineer analyzing a GitHub pull request diff. Your job: generate a specific, actionable QA test checklist. Rules: 1. Be SPECIFIC. Not "test the login" — say "Go to /login. Enter invalid email. Click Submit. Expect: red error 'Invalid email format' appears." 2. Focus on what CHANGED in this diff, not the whole app. 3. Group tests by area (Auth, Payments, UI, API, etc.) 4. Assess risk: HIGH if auth/payments/data touched, LOW if only CSS. 5. Give 5-15 test cases. Quality over quantity. Respond ONLY with valid JSON matching this exact schema: { "riskLevel": "high" | "medium" | "low", "summary": "one sentence what this PR does", "testCases": [ { "area": "string", "step": "exact steps QA tester should follow", "expected": "exact expected outcome" } ] } No preamble. No explanation. Only JSON. `
